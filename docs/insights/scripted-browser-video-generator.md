# 脚本驱动浏览器教程视频生成器产品思考

## 用户问题

用户希望把自然语言教程脚本直接变成可交付的视频，而不是手工打开浏览器、录屏、配音、打字幕、剪辑。这个流程的痛点是步骤多、依赖多、容易反复返工，尤其是产品教程、功能演示和操作说明视频。

## 为什么 CLI-first

CLI 核心让浏览器控制、TTS、字幕和视频合成保持可测试、可复用。未来可以在 Skill 或 UI 层提供更友好的脚本生成体验，但底层仍复用同一条 pipeline，避免把浏览器和视频工具直接耦合进交互层。

主要取舍：CLI 对小白不够友好，但更适合作为稳定内核；小白体验放到后续包装层解决。

## 为什么用 Playwright

Playwright 可以直接控制 Chromium，并使用 `recordVideo` 录制受控浏览器页面。对教程视频来说，MVP 更需要可重复、可自动化的页面操作，而不是系统级录屏的全局捕获能力。

当前实现采用“准备页 + 录制页”两阶段：先执行动作并等待稳定，再录制分段画面，避免网络加载和页面抖动进入正片。

## 为什么用阿里云 TTS

用户指定阿里云 TTS，因此 MVP 接入 DashScope CosyVoice 非实时语音合成 HTTP API。实现保留 `TtsProvider` 接口，后续可替换为其他云厂商、本地 TTS 或测试 fake provider。

这个选择的代价是需要 DashScope API Key、额度和网络可用性；所以失败报告必须清晰，不能写假音频冒充成功。

## 为什么用 ffmpeg

`ffmpeg` 是本地视频合成最稳妥的基础设施：可拼接 clips、合并旁白音频、烧录字幕，并且适合 CLI 自动化。相比接剪映 MCP 或动画框架，ffmpeg 的结果更可控，也更容易单元测试参数生成和错误报告。

## 为什么暂缓 HyperFrames

HyperFrames 更适合做标题卡、动效、包装字幕和高阶视觉表达，不适合在 MVP 阶段承担浏览器自动化和素材合成的核心职责。当前优先证明“脚本 → 浏览器录制 → TTS → 字幕 → MP4”的主链路可跑通。

## 内置 demo 的意义

示例脚本使用 `demo:basic` 内置页面，避免用户为了试跑还要先启动 dev server 或替换端口。它不是最终产品形态，而是降低首次验证成本的教学入口。

## 已知局限

- 字幕样式目前交给 ffmpeg 默认行为，跨平台字体表现还需后续验证。
- Playwright 录制的是受控 Chromium 页面，不覆盖桌面软件或系统级操作。
- 真实 TTS smoke 依赖阿里云凭据、额度和网络状态。
- MVP 还没有 Skill/UI 包装层，自然语言脚本仍需按规则书写。

## 后续方向

- Skill 层：把更自由的自然语言教程转成当前规则脚本。
- UI 层：提供脚本编辑、预览、失败截图查看和一键重跑。
- 包装层：加入标题卡、章节卡、字幕样式、进度高亮和品牌模板。
- 多 Provider：补充本地 TTS 或其他云 TTS，降低阿里云单点依赖。
